# Core Docker Objects

When you interact with the Docker daemon, you are creating, viewing, and managing various "objects." Understanding these four fundamental building blocks is essential for working with Docker effectively.

---

## 1. Images

An **image** is a read-only template containing the instructions necessary for creating a Docker container. It is the blueprint of your application environment.

*   **Composition:** An image packages everything your application needs to run: the application code, the runtime (like Node.js or Python), necessary system tools, libraries, and default configuration settings.
*   **Creation:** Images are usually defined using a `Dockerfile`. A `Dockerfile` is a simple text file containing a list of commands (like `FROM ubuntu`, `RUN apt-get install python`, `COPY . /app`). Each command in the file creates a distinct "layer."
*   **Immutability:** Once an image is built, it cannot be changed. If you need to update your application or patch a security vulnerability in a base library, you edit the `Dockerfile` and build a completely new image.
*   **Analogy:** If Docker were object-oriented programming, the Image is the **Class**.

---

## 2. Containers

A **container** is a runnable, living instance of an image. If the image is the blueprint, the container is the house built from that blueprint.

*   **Lifecycle:** You use the Docker client to `run`, `start`, `stop`, `pause`, and `delete` containers. A container can be spun up from an image in milliseconds.
*   **Isolation:** A container runs as a standard Linux process, but it is heavily isolated from the host machine and from other containers using Linux Kernel Namespaces and Control Groups (cgroups). It believes it is running on its own dedicated operating system.
*   **Ephemerality:** By default, containers are temporary. Any data written generated by the application inside the container is stored in a thin, temporary read-write layer on top of the image. When the container is deleted, that top layer—and all modifications within it—are permanently destroyed. 
*   **Analogy:** Following the OOP analogy, the Container is the instantiated **Object**.

---

## 3. Networks

By design, containers are isolated. However, real-world applications (like a web server needing to talk to a database) require containers to communicate with each other, the host machine, and the outside internet. This is where Docker **Networks** come in.

Docker networking allows you to define isolated communication channels between specific containers without exposing them to the entire host network.

Docker provides several built-in network drivers:

*   **`bridge`:** The default network driver. If you don't explicitly specify a network, containers attach to the default bridge. It allows containers running on the same host to communicate with each other via IP address, while remaining isolated from the host's wider network.
*   **`host`:** This removes network isolation entirely. The container shares the host’s networking stack directly. If you bind to port 80 in a host-networked container, the application is actually binding to port 80 on the host machine itself.
*   **`overlay`:** Used to connect multiple Docker daemons together, allowing containers running on entirely different physical servers to communicate securely over an encrypted network. (Crucial for Docker Swarm or Kubernetes).
*   **`none`:** Disables all networking for a container, keeping it completely offline and isolated.

---

## 4. Volumes

Because containers are ephemeral, any important data (like a database’s actual records, user uploads, or application logs) must be stored somewhere permanent. **Volumes** are Docker’s preferred mechanism for persisting data.

*   **How They Work:** A volume is a designated directory on the host machine that Docker maps directly into a specific folder inside the container. 
*   **Independently Managed:** Unlike a "bind mount" (which relies on the specific host OS folder structure), volumes are managed entirely by the Docker daemon itself (on Linux, usually inside `/var/lib/docker/volumes/`). This makes volumes completely independent of the host system's specific file hierarchy.
*   **Data Survival:** When you delete a container, the volume it was attached to is *not* deleted. The data survives. You can later spin up a brand new container (perhaps built from an updated image) and attach the exact same volume to it, picking up right where the old container left off.
*   **Sharing State:** Multiple containers can mount the exact same volume simultaneously, allowing them to share read/write access to a common pool of persistent data.
